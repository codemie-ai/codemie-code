# Assistant Skill/Agent Generators — `--conversation-id` Fix Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stop the assistant-as-skill/agent generators in codemie-code from producing SKILL.md / agent.md files that invoke `codemie assistants chat` without a `--conversation-id`. The current templates cause cross-topic context bleed and apply-confirmation corruption when calling LLMs orchestrate multi-turn workflows against remote codemie assistants (e.g. brianna).

**Architecture:** Pure template edits across four sibling generators. Each generator's `dedent`-templated output adds (a) a `--conversation-id "<workflow-id>"` argument in every example, (b) a new "Conversation id" section explaining the rule, (c) a new "Single-shot rule" section, and (d) a revised "Instructions" block that tells the calling LLM how to mint and reuse a workflow id and how to verify writes. No runtime / API changes.

**Tech Stack:** TypeScript generators with `dedent` for template strings. Output is plain Markdown SKILL.md / agent.md.

**Note on tests:** Per AGENTS.md rule 2 ("Tests Only On Explicit Request"), no new tests are added. The existing `__tests__/` files exercise file-write paths, not template content. Verification is mechanical (grep for `--conversation-id` in each generator + manual smoke test by running an end-to-end brianna multi-turn workflow against the patched template in Task 6).

**Files:**

**Test-first:** no — pure template edit; verification via grep + Task 6 end-to-end smoke.

- [ ] **Step 1: Replace the `dedent` template body**
  Replace the existing `Instructions` + `Command format` + `Examples` blocks (the entire `dedent` string after `${metadata}`) with the new version:
  ```ts
  return dedent`
    ${metadata}

    # ${name}

    ${description}

    ## Instructions

    1. **Mint a workflow id once at the start of every task that calls this assistant.** Reuse it for every invocation in that task. Suggested patterns:
       - From a shell: \`workflow_id="${slug}-$(date +%Y%m%d-%H%M%S)-$$"\`
       - From an LLM caller: include the related ticket key (e.g. \`${slug}-EPMCDME-12345\`) or a fresh UUID.
    2. **Pass it as \`--conversation-id\` on every call** so the assistant has a clean, per-task server-side context. Do not rely on the implicit \`CODEMIE_SESSION_ID\` env-var fallback — that id is shared across every assistant invocation in your Claude session and causes cross-topic context bleed.
    3. **For state-changing operations (create / update / delete) put the full final payload in one message.** Do not split the work into a "draft" turn followed by a "confirm and apply" turn — if server-side context is lost between turns, the confirmation message itself can be persisted as the resource content.
    4. **After any write, re-fetch the resource and verify the written content matches what you sent.** If it does not match, the call was lost — resend in single-shot form with the full payload.

    **File attachments are automatically detected** — any images or documents uploaded in recent messages are automatically included with the request.

    **ARGUMENTS**: "message"

    **Command format:**
    \`\`\`bash
    codemie assistants chat "${assistantId}" --conversation-id "<workflow-id>" "message"
    \`\`\`

    ## Examples

    **Simple message:**
    \`\`\`bash
    workflow_id="${slug}-$(date +%Y%m%d-%H%M%S)-$$"
    codemie assistants chat "${assistantId}" --conversation-id "$workflow_id" "help me with this"
    \`\`\`

    **With file attachment** (reuse the same workflow id):
    \`\`\`bash
    codemie assistants chat "${assistantId}" --conversation-id "$workflow_id" "analyze this code" --file "script.py"
    \`\`\`

    **With multiple files** (reuse the same workflow id):
    \`\`\`bash
    codemie assistants chat "${assistantId}" --conversation-id "$workflow_id" "review these files" --file "file1.png" --file "file2.py"
    \`\`\`
  `;
  ```
  Note: \`${slug}\` is already in scope via the line `const slug = assistant.slug || sanitizeToSlug(assistant.id) || assistant.id;` at the top of `createSkillMetadata`. Reuse it inside `createSkillContent` by computing it the same way or by lifting it to the top of `createSkillContent`. Pick whichever keeps the diff smaller.

- [ ] **Step 2: Add the slug variable inside `createSkillContent` if missing**
  At the top of `createSkillContent`, add:
  ```ts
  const slug = assistant.slug || sanitizeToSlug(assistant.id) || assistant.id;
  ```
  (Just below the existing `const assistantId = assistant.id;` line.) Required because the template now references `${slug}`.


  Expected: each call references the prior turn correctly. The final ticket's Description matches the refined draft (not the literal text of the confirmation message). No "Please provide the updated Description..." style corruption.

- [ ] **Step 4: Read-back verification**
  Make one more brianna call (still same `--conversation-id`) requesting a verbatim re-read of the freshly-created ticket's Description. Compare to the refined draft. Confirm match.

- [ ] **Step 5: Negative-control sanity check**
  Run one more brianna call **without** `--conversation-id` to confirm the broken behaviour is still reachable (loses or pollutes context). Use a deliberately ambiguous follow-up like "ok proceed" and observe that the assistant cannot identify what to proceed with. This protects against false-positive verification.
